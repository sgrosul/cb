# CI/CD Lab

Практика по отладке CI/CD через ИИ на базе реалистичного GitHub Actions workflow.

## Цели практики

В рамках этой практики нужно:

1. Сгенерировать черновик исправления CI по логам и контексту проекта.
2. Довести pipeline до стабильного зеленого статуса.
3. Провести самопроверку результата и короткую рефлексию по процессу.

## Условие задания

Нужно исправить конфигурацию `.github/workflows/ci-cd.yml`, чтобы все обязательные проверки в GitHub Actions проходили успешно.

В workflow специально заложены ошибки, аналогичные тем, что встречаются в реальных командах.

### Ограничения

- Не добавлять секреты в репозиторий и в workflow в открытом виде.
- Не менять код в `src/` и `test/` для обхода падений CI.
- Править только инфраструктурную часть задания: workflow и связанные с ним настройки.

## Шаги выполнения

1. Подготовьте среду: Node.js 20+ и `npm`.
2. Выполните локальную базовую проверку: `npm ci && npm run verify`.
3. Запустите workflow через `pull request` и откройте первый упавший run.
4. Найдите в логах первопричину (первая конкретная техническая ошибка, а не финальный `exit code`).
5. Сформируйте запрос к ИИ:
   - контекст шага;
   - 20-40 строк лога вокруг ошибки;
   - соответствующий фрагмент workflow.
6. Попросите у ИИ: первопричину, симптомы, минимальный патч, способ проверки.
7. Внесите минимальную правку и повторно запустите pipeline.
8. Повторяйте цикл `лог -> гипотеза -> правка -> перезапуск`, пока workflow не станет полностью зеленым.
9. Подготовьте отчет 5-10 предложений:
   - какие изменения внесены;
   - как проверяли локально и в CI;
   - какие ограничения текущего pipeline остались.

## Критерии сдачи

Работа считается выполненной, если одновременно соблюдены все пункты:

- [ ] Последний запуск workflow в PR имеет статус `Success` по всем обязательным job.
- [ ] Локально выполнено `npm ci && npm run verify` без ошибок.
- [ ] Изменения затрагивают только CI/CD-конфигурацию и не меняют бизнес-логику `src/`/`test/`.
- [ ] В workflow нет открытых секретов, токенов, паролей и иных чувствительных данных.
- [ ] В отчете есть 3 обязательных блока: «что изменено», «как проверено», «какие ограничения остались».
- [ ] Для каждого исправления вы можете указать: первопричину из лога и строку/блок в workflow, который был изменен.

## Чеклист самопроверки

- [ ] Я могу объяснить, почему упал первый запуск CI, в 1-2 предложениях.
- [ ] Я исправлял первопричины, а не скрывал симптомы.
- [ ] После каждой правки делал повторный запуск и сверял результат.
- [ ] Порядок шагов в pipeline логичен и воспроизводим.
- [ ] Артефакты передаются между job корректно.
- [ ] Локальная проверка и проверка в CI дают согласованный результат.
- [ ] Итоговый отчет написан своими словами, без копирования чужого текста.

## Рефлексия

Подумайте над процессом отладки и ответьте на вопросы:

1. Что в вашем решении сработало лучше всего и почему?
2. Что бы вы сделали иначе, если бы начинали заново?
3. Сравните свое решение с эталонным (после разбора с преподавателем): какие отличия и чему они вас научили?

Ошибки в этом задании ожидаемы и нормальны. Цель практики - научиться диагностировать и исправлять их по шагам, а не получить идеальный результат с первой попытки.
